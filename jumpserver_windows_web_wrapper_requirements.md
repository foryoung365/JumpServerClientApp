# JumpServer 社区版 Windows Web 套壳客户端需求整理

## 项目背景

当前使用 **JumpServer 社区版** 管理 **Windows 远程桌面**。  
由于社区版只能使用 **Web 方式访问 Windows**，无法使用企业版的 **RDP 客户端链路**，导致在浏览器中使用远程 Windows 时，出现以下问题：

1. **大量快捷键无法正常使用**
   - 浏览器劫持常见快捷键，例如：
     - `Ctrl+P`
     - `Ctrl+F`
     - `Ctrl+H`
     - `Ctrl+L`
     - `Ctrl+N`
     - `Ctrl+O`
     - `Ctrl+S`
     - `Ctrl+T`
     - `Ctrl+Tab`
     - `Alt+Left / Alt+Right`
     - `F11`
   - 在远程 Windows 中使用 **VS Code** 时，开发体验很差。

2. **输入法状态经常错乱**
   - 本机中英文状态与远端 Windows 状态不一致；
   - 经常出现“本机以为是中文，远端实际上是英文”或反过来的情况；
   - 中文输入体验很差，尤其是编辑器、文档、注释输入场景。

---

## 已确认的前提条件

后续所有方案讨论，必须基于以下前提：

1. **不修改 JumpServer 服务端**
2. **只考虑 JumpServer 社区版**
3. **只考虑 Windows 远程桌面场景**
4. **不考虑企业版 Razor / 原生 RDP Client 链路**
5. **方案必须建立在现有 Web 图形链路之上**
   - 即本质上仍然是 JumpServer 社区版现有的 Web 访问方式
   - 只能从客户端侧做增强

---

## 当前初步预定方案

### 方案方向
开发一个 **桌面套壳客户端**，本质上是对 JumpServer 的 **Windows Web 访问页面** 进行桌面化封装。

可选技术方向示例：
- Electron
- Tauri
- 其他具备 WebView + 键盘事件拦截能力的桌面壳

### 该方案的定位
它不是原生 RDP 客户端，也不是重做协议栈。  
它的目标是：

- **减少浏览器层面对快捷键的劫持**
- **优化输入焦点管理**
- **改善远程 Windows 中 VS Code 的常用快捷键体验**
- **通过本地输入策略缓解中文输入法错位问题**

---

## 预期实现目标

### 目标 1：改善快捷键体验
希望尽量恢复或改良以下类型快捷键的可用性：

#### 优先级最高：VS Code / 编辑器常用快捷键
- `Ctrl+C`
- `Ctrl+V`
- `Ctrl+P`
- `Ctrl+Shift+P`
- `Ctrl+F`
- `Ctrl+H`
- `Ctrl+B`
- `Ctrl+Shift+F`
- `Ctrl+/`
- 其他与开发高频相关的快捷键

#### 可接受方式
不要求所有快捷键都“物理直通”，可以接受：
- 本地拦截后转发
- 改映射
- 提供菜单/按钮发送特殊组合键

### 目标 2：缓解中文输入法错乱
希望重点改善以下问题：

- 本机与远端中英文状态不一致
- 中文输入候选框与实际状态错乱
- 远端输入中文时体验极差
- VS Code/文档场景下输入中文注释、文字内容很痛苦

理想目标：
- 中文正文输入尽量基于**本地输入法**完成，再传给远端
- 减少对远端输入法状态的依赖

### 目标 3：保留现有 JumpServer 使用方式
希望客户端仍然基于现有 JumpServer 流程使用，不破坏：
- 登录流程
- 资产访问流程
- 社区版现有 Web 访问能力

---

## 明确的非目标

以下内容**不在当前方案目标内**：

1. **不追求原生 mstsc 级别体验**
2. **不追求完整支持所有系统级快捷键**
3. **不追求替代 JumpServer 企业版的 Razor 能力**
4. **不改 JumpServer 服务端，不新增服务端接口**
5. **不自研原生 RDP 客户端**
6. **不解决所有输入问题，只优先解决中文正文输入和高频快捷键问题**

---

## 已达成的关键判断

### 1）套壳方案能解决什么
在不改服务端的前提下，套壳方案**有价值**，主要能解决：

- 浏览器菜单/标签页/地址栏导致的快捷键冲突
- 部分 `Ctrl` 类快捷键被浏览器劫持的问题
- 输入焦点不稳定的问题
- 中文正文输入体验差的问题（前提是采用本地 IME / 文本输入策略）

### 2）套壳方案解决不了什么
它**不能彻底解决**：

- `Win+R`
- `Win+X`
- `Alt+Tab`
- `Ctrl+Alt+Del`
- 其他被操作系统级别保留的快捷键
- 原生 RDP 级别的完整按键语义
- 本机和远端输入法状态的真正“同步”

### 3）输入法问题的判断
如果只是单纯做“Web 套壳”，输入法问题**不会自动消失**。  
要改善这一点，必须引入**双模输入策略**：

- **普通按键模式**：用于快捷键、编辑器控制、终端操作
- **本地 IME / 文本输入模式**：用于中文正文输入

这样才能在不改服务端前提下，显著缓解输入法错乱。

## 建议的客户端能力范围

### 第一阶段最小可行版本
希望先验证以下能力：

1. 打开并承载 JumpServer 的 Windows Web 页面
2. 去掉浏览器层 UI 干扰
3. 全屏 / 沉浸模式
4. 稳定焦点管理
5. 对常见高频快捷键做本地拦截与转发
6. 提供少量特殊按键发送能力
7. 初步支持中文输入优化策略

### 第二阶段增强方向
在 MVP 验证有效后，再考虑：

1. 快捷键映射配置
2. 双模输入切换 UI
3. 发送特殊按键面板
4. 日志与问题诊断
5. 多环境/多 JumpServer 地址配置
6. 登录态保持
7. 打包与分发

## 需要 Codex 重点讨论的问题

请基于以下前提继续讨论和设计：

### A. 技术可行性
- 在**不修改 JumpServer 服务端**前提下，桌面套壳客户端如何实现最大化收益？
- 更适合选 Electron、Tauri，还是其他技术栈？
- 对 JumpServer Web 页面做“纯承载”更合适，还是需要注入脚本增强？

### B. 快捷键策略
请重点分析：
- 哪些快捷键适合直接透传
- 哪些快捷键适合本地拦截后改映射
- 哪些快捷键注定无法支持，只能做按钮/菜单发送
- 针对 VS Code，建议一套适合远程 Web 场景的快捷键处理策略

### C. 输入法策略
请重点分析：
- 如何在不改服务端前提下，实现“本地中文输入优先”
- 如何在“快捷键模式”与“文本输入模式”之间切换
- 哪些输入场景应走本地 IME 文本输入
- 哪些场景必须保留真实按键透传

### D. MVP 设计
请给出一个最小可行版本设计：
- 客户端架构
- 关键模块
- 主要风险
- 第一版应优先实现哪些能力
- 哪些功能应明确推迟
